CLI parity. Every control maps 1:1 to a dbtool subcommand/flag, including the new releases and rollback-to-release <VERSION> from PR #471 (plus the existing migrate, apply, rollback-to, status, --dry-run, --no-lock, --jobs, --plugins-dir, --connection-string, --schema). The GUI reuses Lightweight::SqlMigration::MigrationManager directly — no duplicate logic.
Release model (faithful to PR #471). Releases are {version, highestTimestamp} pairs declared in plugin TUs via LIGHTWEIGHT_SQL_RELEASE(version, timestamp). MigrationManager merges them from every loaded plugin — there's no schema_releases table, no yaml, no git tags. Per-release status in the UI (applied/partial/pending) is computed exactly like dbtool releases does, via GetMigrationsForRelease + GetAppliedMigrationIds. The "unreleased" group covers migrations whose timestamp is greater than every registered release.
Reusable QML component. The center list + log panel can ship as Lightweight.Migrations 1.0 / MigrationView with a migrationManager QObject property exposing allReleases, migrationsForRelease(v), and the existing apply/rollback APIs. Downstream apps drop it in with import Lightweight.Migrations 1.0 and bind their own MigrationManager instance (same one they already feed to dbtool).
